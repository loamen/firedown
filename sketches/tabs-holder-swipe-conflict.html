Tabs holder — swipe-to-close vs ViewPager swipe

Two horizontal-swipe gestures live in the same coordinate space on this screen. The ViewPager swiping Tabs ↔ Incognito and the per-tile swipe-to-close both want the same drag, so they race the system's onInterceptTouchEvent chain — ViewPager2 wins when the drag has any horizontal component, which means swipe-to-close only fires reliably if the user starts dragging from the card center with no vertical drift.

What's happening

Two horizontal-swipe handlers are attached in the same view tree:

  • ViewPager2 at the root, setUserInputEnabled(true), swaps Tabs ↔ Incognito
  • ItemTouchHelper on each tile's RecyclerView, LEFT | RIGHT swipe directions, calls onSwiped → close

ViewPager2 intercepts touches that look like horizontal scrolls at its own onInterceptTouchEvent layer, before ItemTouchHelper sees them. So:

  • Slow / careful horizontal drag from the tile centre → ItemTouchHelper wins → close
  • Any drag that has even slight horizontal velocity / drift → ViewPager2 grabs it → page switch
  • Effect: swipe-to-close feels broken / unreliable; users default to the × button instead

Both gesture targets compete for the dashed zone above. The × button on each tile already covers the close action reliably — swipe is the redundant one in the pair.

The fix

Disable ViewPager2's built-in swipe. The segmented Tabs / Incognito selector at the top already covers mode switching — it's the affordance the user actually looks at. Tap-only switching is the dominant pattern in modern Android (Gmail, Drive, Files).

// TabsHolderFragment.setupViewPager() mViewPager.setAdapter(new TabsPagerAdapter(this)); mViewPager.setOffscreenPageLimit(1); mViewPager.setUserInputEnabled(true); mViewPager.setUserInputEnabled(false); // tap-only via segmented selector

Wins

  • Swipe-to-close on tab tiles now fires reliably — no upstream interceptor
  • Mode-switch UX is unchanged for users who tap (the dominant path)
  • One-line change, no layout or adapter touched

Trade-offs

  • Users who learned the swipe-to-switch-mode gesture lose it. The segmented buttons stay; muscle memory shifts to taps.
  • The ViewPager2 backing graph stays in place (still hosting the two fragments) — we only suppress its input handling, not the page model. Cheap and reversible.

Why drop ViewPager swipe instead of dropping swipe-to-close?

Either gesture would resolve the conflict. The argument for keeping swipe-to-close over swipe-to-switch:

  • Frequency mismatch. Closing tabs is something users do dozens of times a session. Switching between Regular and Incognito tab modes is something a privacy-conscious user might do a few times a day at most.
  • Discoverability. The segmented selector is a visible affordance for mode switching. The × button on each tile is the visible affordance for close. The hidden gestures (swipe-to-close and swipe-to-switch) are accelerators — swipe-to-close is the one users actually internalise because it matches every other list in Android (Gmail archive, notification dismiss).
  • Symmetry. Once swipe-to-close is reliable, the per-tile × button can stay as the visible affordance for newer users; the swipe becomes the accelerator for users who learn it. Two paths, no conflict.
