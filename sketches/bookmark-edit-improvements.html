Edit bookmark — three directions

Today's screen is two text fields gated behind an "Edit / Save" two-tap pattern, with a plain text-button Delete at the bottom. Below: today as baseline, a minimal fix that just removes the friction without redesigning anything, and a refined version that brings the screen into the Firedown card vocabulary with an identity header, overflow actions, and a destructive-card Delete that matches the popup's Quit treatment.

Today — baseline

Friction points

  • "Edit" → "Save" two-tap pattern with no benefit
  • Delete reads as a passive text link, equal weight to "Edit"
  • No visual anchor — both fields read as empty on entry
  • No "test the URL" or "share" affordance

A — Minimal fix

Smallest change that resolves the real friction. Fields are editable on entry. Save button always says "Save", enabled when title + URL are non-empty. Delete adopts the error-container chip + error text treatment from the popup's Quit row.

My pick

A first, then B on top of it. A is the actual UX bug fix — the Edit/Save two-step is friction and the plain Delete is wrong. Should land in one small commit and resolve the real complaint. B is a polish layer that brings the screen into the Firedown card vocabulary we've been establishing across Home and the popup; worth doing but doesn't have to be the same commit.

If you want one shot, do B — the visual cost is the same to read either way once you're committed to a layout change.

What I'd hold off on: folder support (only worth it if the data model already has folders), confirm-on-back for unsaved changes (real risk, but minor), inline URL validation (current "prepend https://" is fine for now).
