More popup — reconsidered with the real sheet structure

The Browser popup is not a flat list — it has a top quick-row of page-context actions (Back · Forward · Share · Refresh) and a vertical list below, with one item (Desktop site) already a stateful toggle. That's the right scaffold; the problem is which items live in which row, and which item types belong in this sheet at all.

What I had wrong in v2: I proposed "add Share" — Share is already in the top quick-row. I drew the popup as a flat sectioned list — it's actually quick-row + list. And I underread that Desktop site is already a toggle, which means the popup is comfortable hosting stateful page controls — Bookmark page should join that pattern instead of pretending to be a destination.

The quick-row is already where page-context actions live (Back, Forward, Share, Refresh). Bookmark page belongs there too — it's a one-tap toggle on the current page, identical pattern to Share/Refresh. Becomes a star icon that fills when bookmarked. Why this is the right primitive

The quick-row is a state-toggle row by nature (Refresh fires-and-resets; Back/Forward state-driven; Share is the one true action). Putting Bookmark there as a state toggle (fills when bookmarked) matches that grammar — it stops being "a button that says 'Bookmark page' that doesn't actually navigate anywhere", which is a confusing read next to "Bookmarks" which DOES navigate.

  • Glance-able: ★ tells you instantly whether the current page is bookmarked
  • Resolves the "Bookmark page" vs "Bookmarks" word-collision in the list
  • Quick-row stays at 5 icons — comfortable, still room for Desktop site if you want to promote that too
  • Home popup becomes 3 + Quit = real overflow
  • Bookmark on the quick-row may not include "edit folder / name" — that flow stays in the bookmarks list. May need a long-press on ★ to open the edit dialog.
  • Five icons on the quick-row at small phone widths can get tight — Galaxy A-class phones in the 360dp range will need labels to ellipsize or shrink.

D2 — Promote both stateful actions, shrink the list further

If you commit to the quick-row as the state-toggle row

D1's logic, extended. Desktop site is also a stateful page toggle — same shape as Bookmark page. Put both in the quick-row and the list becomes pure destinations: Bookmarks · Find · History · Safe Folder · Settings. Cleaner mental model: top row = state of THIS page · list = places to go. The list now has zero stateful items, zero word-collisions, and a clear job.

Why and the risk

This is D1 with the rule applied consistently. Mental model becomes sharp: top row = "what's this page doing right now", list = "take me somewhere". Five list items, six quick-row icons.

  • One rule: stateful = top row, navigational = list
  • List has zero collisions ("Bookmarks" no longer next to "Bookmark page")
  • Desktop site as a top-row icon (with active-state tint) is more glance-able than a buried toggle row
  • Six quick-row icons gets cramped on narrow phones — at 360dp each gets ~58dp, labels may need to shrink to 9px or wrap
  • Desktop site's on/off label ("View as Desktop") is more readable than a tinted icon — losing the explicit text is a minor accessibility regression

My actual pick

D1. Move "Bookmark page" up to the quick-row as a star toggle, leave Desktop site in the list as the explicit-label toggle it already is, and dedupe the Home list. Five quick-row icons fits comfortably; the list keeps its one toggle (Desktop site is the one place an explicit "on/off" label is worth having since the state name reads differently from the icon meaning).

Why not D2: six icons on the quick-row is genuinely tight on the phones your target audience runs, and "Desktop" as a tinted icon loses the readable on/off state. D1 captures 90% of the clarity win at zero space risk.

Net popup contents I'd commit to:

  • Browser quick-row: Back · Forward · ★ Bookmark · Share · Refresh
  • Browser list: Bookmarks · Find in page · Desktop site (toggle) · History · Safe Folder · Settings
  • Home popup (list-only, no quick-row): History · Settings · Quit
